# Brevis Vera - Presentation Guide

## Project Overview

**Brevis Vera** is a zero-knowledge proof (ZKP) system for verifiable image editing with C2PA (Coalition for Content Provenance and Authenticity) provenance. It allows users to prove that image edits were performed correctly without revealing the original image or edit parameters.

## Problem Statement

When images are edited (cropped, resized, brightness adjusted, etc.), how can we verify:
1. The original image had a valid C2PA provenance certificate
2. The edit operations were applied correctly
3. The output image is the result of those operations

Traditional approaches require revealing the original image or trusting a central authority. Brevis Vera uses zkVM proofs to verify everything privately.

## Key Components

### 1. **lib** (C2PA Library)
- Extracts C2PA manifests from JPEG/PNG images
- Parses manifest store, claims, and certificate chains
- Verifies C2PA signatures using the official `c2pa` crate
- Provides metadata: issuer, timestamp, signature, claim_hash

### 2. **app-lib** (Shared Library)
- Defines image operation types (Crop, Resize, Brightness, etc.)
- Contains zkVM guest code for applying operations
- Computes semantic hash of processed images
- Provides `PublicValuesStruct` for Solidity verification

### 3. **prover** (Host Application)
- Orchestrates the proof generation process
- Extracts C2PA from original image
- Applies image operations using `image-editor` crate
- Computes file hash of output image
- Sends data to zkVM for proof generation
- Uses pico-sdk for zkVM proof generation

### 4. **verifier** (CLI)
- Validates proofs generated by the prover
- Checks proof against known constraints
- Returns: is_valid, final_image_hash, num_operations

### 5. **image-editor** (Host-side Image Processing)
- Rust library for actual image manipulation
- Supports: crop, resize, brightness, contrast, exposure, gamma, thumbnail

## Supported Image Operations

| ID | Operation | Parameters |
|----|-----------|------------|
| 0 | Crop | x, y, width, height |
| 1 | Resize | width, height |
| 2 | ResizeAspect | max_width, max_height |
| 3 | Brightness | value (i32) |
| 4 | Contrast | value (i32) |
| 5 | Exposure | value (f32) |
| 6 | Gamma | value (f32) |
| 7 | Thumbnail | max_size |
| 8 | CropResize | x, y, width, height, target_width, target_height |

## Data Flow

```
1. INPUT IMAGE (with C2PA manifest)
         │
         ▼
2. C2PA EXTRACTION (lib)
   - Parse manifest store
   - Extract issuer, timestamp, signature, claim_hash
   - Verify certificate chain
         │
         ▼
3. IMAGE OPERATIONS (host-side)
   - Apply operations to generate output image
   - Compute file hash of output
         │
         ▼
4. zkVM PROOF GENERATION
   - Send to zkVM:
     • C2PA metadata (as private inputs)
     • Original image (downscaled)
     • Operation list
     • Final image hash (expected)
   - zkVM applies operations internally
   - zkVM computes semantic hash
   - zkVM outputs: final_image_hash, num_operations
         │
         ▼
5. PROOF OUTPUT
   - Proof bytes
   - Public values: final_image_hash, num_operations
```

## Key Innovation: Dual Computation

The system ensures correctness by computing the image hash in two places:

1. **Host (Prover)**: Applies operations using `image-editor` crate, computes SHA256 of output file
2. **zkVM (Guest)**: Applies operations internally on downscaled image, computes semantic hash

If these match, the proof is valid - proving that operations were applied correctly without revealing parameters.

## Privacy Features

- **Operation parameters**: Private inputs to zkVM (not revealed in proof)
- **Original image**: Only a downscaled version (max 200x200) sent to zkVM
- **C2PA metadata**: Can be kept private or revealed as needed
- **Public outputs**: Only `final_image_hash` and `num_operations`

## Verification (Solidity)

```solidity
struct PublicValuesStruct {
    bytes32 final_image_hash;
    uint32 num_operations;
}
```

On-chain verification checks that the proof is valid and the public values match expectations.

## Running the Project

### Build
```bash
cargo build --release
```

### Generate Proof
```bash
# Crop and resize
cargo run --release --bin prover -- \
    image.jpg \
    --op crop,100,100,800,600 \
    --op resize,400,300 \
    -o output.jpg

# Thumbnail
cargo run --release --bin prover -- \
    image.jpg \
    --op thumbnail,200 \
    -o thumb.jpg
```

### Verify Proof
```bash
cargo run --release --bin verifier
```

## Use Cases

1. **Content Authenticity**: Prove an image originated from a C2PA-certified camera
2. **Edit Verification**: Prove edits were applied correctly without revealing the image
3. **Supply Chain**: Track image provenance through editing pipelines
4. **Social Media**: Verify image authenticity without compromising privacy

## Technical Stack

- **Language**: Rust
- **ZK Framework**: Pico (risc0-based)
- **C2PA**: Official `c2pa` crate
- **Image Processing**: `image` crate + custom `image-editor`
- **Serialization**: serde, serde_cbor, bincode
- **Cryptography**: sha2, x509-parser, alloy-sol-types

## Demo Flow

1. User opens web frontend (image-editor-frontend)
2. User uploads C2PA-enabled image
3. User selects edit operations (crop, resize, etc.)
4. Frontend sends request to server
5. Server runs prover to generate ZK proof
6. Proof and output image returned to frontend
7. User can verify the proof on-chain

## Security Guarantees

1. **Soundness**: If proof verifies, operations were definitely applied
2. **Zero-Knowledge**: Original image and parameters remain private
3. **C2PA Integration**: Provenance verified via standard C2PA certificates
4. **Dual Computation**: Hash mismatch = proof rejection
